# Company Intelligence Dossier — Design

**Date:** 2026-04-13
**Status:** Approved
**Scope:** A per-company dossier generated from job-posting signals (cached job description + URL signals + extracted company name) and surfaced as a new tab in the application expanded row. Pro-gated on generation; Free users can read any dossier another user has already generated.

## Goal

When a user opens an application, show a one-paragraph summary of the hiring company plus a handful of key facts — without leaving Snag, without reading the full JD twice. Compound value across users: the more companies get snagged, the richer the shared cache becomes.

## Non-Goals

- Scraping Crunchbase / LinkedIn / Glassdoor / news feeds (fragile, paid, or legally fraught).
- Fetching the company's external homepage (adds another HTTP hop + scraping concerns). V1 works purely from the JD we already cache.
- Tracking news updates over time.
- Warm-intro finder (separate future spec).

## Data Model

### Migration `010_company_dossiers.sql`

```sql
BEGIN;

CREATE TABLE IF NOT EXISTS company_dossiers (
  id                    UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  company_key           TEXT NOT NULL UNIQUE,
  display_name          TEXT NOT NULL,
  source_domain         TEXT,
  summary               TEXT,
  facts                 JSONB NOT NULL DEFAULT '[]'::jsonb,
  links                 JSONB NOT NULL DEFAULT '{}'::jsonb,
  generated_by_user_id  UUID REFERENCES users(id) ON DELETE SET NULL,
  tokens_in             INT NOT NULL DEFAULT 0,
  tokens_out            INT NOT NULL DEFAULT 0,
  created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_company_dossiers_key ON company_dossiers(company_key);

COMMIT;
```

`company_dossiers` is intentionally **tenant-independent** — it's public company info. No `tenant_id` column. FK only to `users` for attribution (set null on user deletion).

### Company Key Normalization

Helper `companyKey(company, sourceUrl)` in `src/services/dossier.js`:

```js
function companyKey(company, sourceUrl) {
  // Prefer the URL-derived slug (from extractCompanyFromUrl in anthropic.js)
  // because it's more deterministic than user-typed company names.
  const fromUrl = extractCompanyFromUrl(sourceUrl || '');
  const base = (fromUrl || company || '').toLowerCase().trim();
  return base
    .replace(/[.,]/g, '')
    .replace(/\b(inc|llc|corp|co|company|ltd|limited)\b/gi, '')
    .replace(/[^a-z0-9]+/g, '-')
    .replace(/^-+|-+$/g, '')
    .slice(0, 80);
}
```

Same company accessed via different postings converges to the same key. Examples:
- "Machinify Inc." / "machinifyinc" (Greenhouse slug) → both normalize to `machinify`.
- "Acme Corp." / "Acme" → `acme`.

## Dossier Content Shape

Generated by one Sonnet call given the cached JD text, company name, and source URL. Returns JSON:

```json
{
  "summary": "One paragraph (2-4 sentences). What the company does, who they serve, notable stage/funding/size signal if detectable.",
  "facts": [
    "Industry: Healthcare AI / claims automation",
    "Stage: Series B (~$50M raised)",
    "Size signal: ~100-250 employees",
    "Notable products: Pay Optimizer, Claim Insights",
    "HQ: Palo Alto, remote-friendly"
  ],
  "links": {
    "website": "https://machinify.com",
    "linkedin": "https://linkedin.com/company/machinify"
  }
}
```

- `summary` — 2-4 sentence narrative paragraph.
- `facts` — array of 3-6 concise bullet strings. Format: `"Label: value"`.
- `links` — optional. Only populated when the JD text explicitly mentions URLs. Never guessed.

Sonnet is instructed to mark unknowns with `"unknown"` rather than hallucinate. The prompt explicitly says: if the JD doesn't mention funding stage, don't invent one; if size isn't inferable, say "Size: unknown."

## Architecture

### Service (`src/services/dossier.js`)

Exports:

```js
companyKey(company, sourceUrl) → string
getCachedDossier(key) → dossier | null
buildDossier(userId, company, sourceUrl, jdText) → dossier    // expensive; writes to DB
isFresh(dossier) → bool                                        // < 30 days old
```

`buildDossier` performs:
1. Derive `company_key`.
2. Load JD text (caller passes it; usually `app.jd_text` or re-fetched via `fetchJobDescription`).
3. One Sonnet call with a strict JSON prompt.
4. Parse, validate shape, truncate strings.
5. Upsert into `company_dossiers`.
6. Return the row.

Throws only on validation failure; HTTP errors bubble.

### Routes (`src/routes/dossier.js`)

Mounted under `/api`.

- `GET /applications/:id/dossier` → returns `{ dossier, cached, stale, quota }` for the current user. If no dossier exists and the app has a `source_url` or `company`, it does NOT auto-trigger — it returns `{ dossier: null, quota: {...} }` so the UI can render the "Build" CTA with the remaining quota.
- `POST /applications/:id/dossier/build` → synchronously generates (or re-generates) the dossier for that app's company. Checks quota first (Free = 3/week of NET-NEW generations; cached hits don't count). Returns `{ dossier, quota }`.
- No DELETE endpoint in V1 (user can't clear a shared cache).

### Background trigger

In `src/routes/applications.js`, the existing `autoSelectResumeInBackground` pattern kicks in on application creation. Add a sibling `autoBuildDossierInBackground(tenantId, userId, app)`:

- Fires on POST /applications and POST /job-board/snag.
- Skips if the user is Free AND quota already exhausted.
- Skips if a fresh cached dossier already exists for the company_key.
- Skips if `ANTHROPIC_API_KEY` isn't set.
- Never blocks the response.

This is the "magic" that makes dossiers feel automatic — most of the time by the time the user clicks into the row, it's already there.

### Gating

Uses the existing `middleware/tier.js` pattern. Add a new quota entry:

```js
const LIMITS = {
  cover_letters_per_week: 3,
  resumes: 1,
  dossier_generations_per_week: 3,  // new
};
```

Generation events are logged to `usage_log` with `action = 'dossier_generation'`. The quota check uses the existing `checkAiLimit('dossier_generation')` middleware on `POST /applications/:id/dossier/build`.

Reading (`GET /.../dossier`) is always free. Background auto-build is silent when quota is exhausted (no error shown).

### Event Logging (F1 integration)

New event types:
- `company_dossier.built` — payload: `{ company_key, from_cache: false, tokens_in, tokens_out, jd_length_bucket }`
- `company_dossier.read` — payload: `{ company_key, from_cache: true, stale: bool, age_days }`
- `company_dossier.refresh_requested` — payload: `{ company_key, was_stale: bool }`

Instrumented in the dossier service.

## Frontend

### New tab in `ApplicationRow.jsx`

Add a "Company" tab to the ExpandedDetail tab bar:

```jsx
[
  ['timeline', 'Timeline'],
  ['company', 'Company'],     // NEW — position after timeline
  ['notes', 'Notes'],
  ['people', 'People'],
  ['materials', 'Materials'],
  ...(app.status === 'interviewing' ? [['interview', 'Interview Prep']] : []),
]
```

New component `CompanyTab` (in the same file):

- Fetches `/applications/:id/dossier` via `useQuery`.
- **If cached & fresh:** render summary + facts + links + small "Refreshed N days ago" line + optional Refresh button (Pro only).
- **If stale (> 30 days):** render the cached content with an inline amber banner "Dossier is over a month old — Refresh (Pro)" or "…— Upgrade to refresh."
- **If no dossier & user has quota:** render "Build dossier" button (calls POST /dossier/build).
- **If no dossier & Free user with no quota:** render locked state with upgrade CTA.
- **While building:** show spinner state; on success, invalidate the query so the dossier renders.

Layout is a stacked reader view — summary paragraph at top, facts as a vertical list below (each with a small tag color), links as a row of pill links at the bottom.

### Settings — no changes needed in V1.

## Cost Envelope

Per dossier generation: ~5-10K input tokens (JD + signals) + ~500 output tokens. At Sonnet 4.6 pricing ≈ $0.025. Cached 30 days + shared across tenants.

Back-of-envelope: if Snag has 1K users averaging 50 unique-to-cache companies each but with heavy overlap (most snag from the same top-500 companies), net cache population = ~3K dossiers in month 1. That's ~$75.

## Error Handling

- Sonnet returns non-JSON → log, return 500 to the explicit build endpoint; auto-build swallows silently.
- Sonnet returns JSON without required fields → log, attempt to salvage (use summary if present, empty facts), store whatever's valid.
- JD text is empty or very short (<200 chars) → skip generation; return 422 from build endpoint with `{ error: 'Not enough context' }`.
- `source_url` missing and `company` field is empty → refuse to build with a helpful message.
- Quota hit on build endpoint → standard 429 with `upgrade: true` (matches cover letter quota behavior).

## Testing

Manual smoke post-deploy:

1. Create a fresh application with `source_url` set. Within a few seconds, refresh the expanded row → Company tab shows a populated dossier.
2. Verify `product_events` has `company_dossier.built` with `from_cache: false`.
3. Delete the app; create another app for the same company. Tab should show the cached dossier INSTANTLY and emit `company_dossier.read` with `from_cache: true`.
4. As a Free user, manually generate 4 dossiers for net-new companies. Fourth should hit the weekly cap; UI shows locked state.
5. Pro user on a stale dossier → Refresh button works; timestamp updates.
6. Upload a PDF-less app (no source_url, typed "Fake Company Name") → Company tab shows "Not enough context" gracefully.

## Open Questions Resolved

- **Tenant-scoped vs global dossiers?** Global — company info is public. Attribution via `generated_by_user_id` for debugging.
- **Auto-build on create vs lazy?** Auto-build on create when quota allows. Lazy build is the fallback when auto is gated or skipped.
- **Refresh UI for Free users?** Read-only. Upgrade CTA replaces the refresh button.
- **30-day TTL hardcoded?** Yes for V1. Adjustable later if companies meaningfully drift faster.
